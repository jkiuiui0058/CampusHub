# 架构候选方案对比表
（文档要求任务1.2.1：要求 AI 对比多种方案的优劣的输出）

## 1. 分析前提

本表用于完成 Phase 2 中“候选架构分析”任务，先客观说明常见架构方案的特点，再比较各方案的优劣，并结合 CampusHub 当前项目情况分析适配度。

当前仓库中能够确认的项目事实如下：

- 项目是一个校园互助服务平台，核心业务包括需求发布、接单、订单流程管理、评价与信用体系。
- 前端技术栈为 Vue 3 + Vite。
- 后端技术栈为 Spring Boot + Maven，数据库规划为 MySQL（暂未接入）。
- 后端 README 已给出 `controller`、`service`、`mapper` 等典型分层目录，说明当前项目骨架天然接近分层式 Web 应用。
- 当前阶段仍以项目框架搭建为主，尚未进入复杂分布式系统建设阶段。

因此，本对比不仅关注“理论上是否先进”，还重点关注：

- 是否适合课程项目当前阶段；
- 是否方便小团队协作开发；
- 是否利于后续新增功能和接口演进；
- 是否能在 10 周课程周期内稳定落地。

---

## 2. 常用架构方案的客观特点

| 架构方案 | 客观特点 | 常见适用场景 |
|------|------|------|
| MVC | 将系统分为 Model、View、Controller，强调界面、流程控制、数据处理分离 | 中小型 Web 应用、教学项目、快速开发场景 |
| 分层架构 | 按表示层、业务层、数据访问层等进行分层，不同层职责清晰 | 企业常见 Web 系统、前后端分工明确的项目 |
| 微服务架构 | 按业务能力拆成多个独立服务，各服务可独立开发、部署、扩展 | 业务复杂、团队规模较大、需要独立扩缩容的系统 |
| 事件驱动架构 | 模块之间通过事件而非直接调用协作，强调异步、解耦和可扩展性 | 通知、消息处理、异步任务较多的系统 |

---

## 3. 各候选方案优劣与 CampusHub 适配度对比

| 对比维度 | MVC | 分层架构 | 微服务架构 | 事件驱动架构 |
|------|------|------|------|------|
| 核心组织方式 | 按 Model / View / Controller 分工 | 按表示层 / 业务层 / 数据访问层分工 | 按业务领域拆成多个独立服务 | 按事件发布与订阅组织协作 |
| 开发复杂度 | 较低，结构直观 | 较低到中等，规范清晰 | 高，需要拆服务、定义通信方式、处理部署问题 | 中到高，需要设计事件流与异步处理逻辑 |
| 可维护性 | 中等，项目变大后 Controller 易臃肿 | 高，职责边界清楚，便于定位问题 | 理论上高，但前提是服务边界划分合理 | 中等，解耦较好，但问题排查较复杂 |
| 可扩展性 | 中等，适合逐步增加功能 | 高，适合在现有层次中新增模块 | 很高，适合服务独立扩展 | 高，适合增加异步扩展能力 |
| 性能 | 对课程项目通常足够 | 对课程项目通常足够 | 小项目中不一定更优，服务间通信反而有额外开销 | 在异步任务场景下表现较好，但主链路设计更复杂 |
| 团队学习成本 | 低 | 低 | 高 | 高 |
| 10 周内可行性 | 高 | 高 | 低 | 中低 |
| 与当前仓库结构匹配度 | 中等。后端使用 Spring Boot，MVC 思想可自然落地，但仓库更明显体现的是“分层”而不是完整 MVC 文档化表达 | 很高。后端已有 `controller/service/mapper` 结构，前后端分离也适合分层设计 | 低。当前仓库只有一个前端和一个后端工程，没有多服务部署和治理基础 | 中低。当前项目还没有消息中间件和异步处理基础设施 |
| 适合作为 CampusHub 当前阶段主架构吗 | 可以作为分析候选，但更适合作为“设计思想”而不是最终表述重点 | 最适合。既符合现有代码骨架，也适合课程周期与团队能力 | 不适合作为当前阶段主方案，更适合写入“未来规模扩大后的演进方向” | 不适合作为当前阶段主方案，但可在后期局部引入做通知、消息提醒等辅助能力 |

## 4. 结合具体协作开发场景分析适配度

### 场景一：新增功能

以“新增失物招领功能”或“新增活动组队功能”为例，需要同时增加前端页面、后端接口、业务逻辑和数据库表设计。

| 架构方案 | 适配度分析 |
|------|------|
| MVC | 可以较快实现，适合在现有控制器中增加入口并补充对应页面与数据处理逻辑；但如果新增功能越来越多，Controller 层容易堆积过多职责。 |
| 分层架构 | 最适合当前项目。新增功能时可以按“前端页面 → controller → service → mapper → 数据库”稳定推进，职责边界清晰，成员之间容易并行分工。 |
| 微服务架构 | 理论上可把新功能拆成独立服务，但对当前项目来说需要额外处理服务拆分、接口通信、部署配置，成本远高于功能本身。 |
| 事件驱动架构 | 如果新增功能本身以异步流程为主会有优势，但 CampusHub 当前多数核心功能仍是典型请求-响应式业务，直接采用事件驱动会让开发复杂度明显上升。 |

结论：在“新增功能”这个最常见的协作开发场景下，分层架构最符合 CampusHub 当前阶段需求。

### 场景二：模块之间调用接口

以“用户接单后，订单模块更新状态，同时评价模块后续可读取订单结果”为例。

| 架构方案 | 适配度分析 |
|------|------|
| MVC | 模块间协作通常仍通过同一个后端应用内部调用完成，实现简单，但随着业务变复杂，逻辑可能集中在少数控制器或服务中。 |
| 分层架构 | 可以把接口调用责任主要放在 service 层，controller 负责接收请求，mapper 负责数据访问，调用路径清晰，便于后续维护。 |
| 微服务架构 | 不同模块如果拆成不同服务，接口调用会更独立，但也会引入网络通信、接口版本管理、服务注册等额外问题。对目前 CampusHub 来说，这些成本偏高。 |
| 事件驱动架构 | 适合“订单完成后自动触发通知或积分更新”这类异步联动场景，但如果核心业务大量依赖事件传播，调试和追踪会更困难。 |

结论：对于当前阶段的接口调用，分层架构兼顾了清晰性和实现成本；事件驱动更适合未来局部处理“通知、提醒、统计”类异步联动需求。

### 场景三：多人协作开发

以 4 名成员并行开发为例，一部分人负责前端页面，一部分人负责后端接口和业务逻辑。

| 架构方案 | 适配度分析 |
|------|------|
| MVC | 上手成本低，适合教学和入门；但在多人并行时，如果控制器职责划分不够清楚，容易出现多人修改同一位置的问题。 |
| 分层架构 | 更适合小团队协作。可以按层次和业务模块分工，例如前端成员负责页面与接口联调，后端成员分别负责 controller/service/mapper，不容易混乱。 |
| 微服务架构 | 表面上适合“按服务分工”，但前提是团队具备足够的服务拆分经验和运维能力。对课程团队来说，学习和协作成本过高。 |
| 事件驱动架构 | 如果团队经验不足，成员往往难以快速理解“事件从哪里发出、被谁消费、出了问题如何排查”，协作门槛较高。 |

结论：在多人协作开发场景下，分层架构的沟通成本最低，最有利于按职责边界组织任务。

## 5. 综合判断

### 5.1 各方案总体优劣总结

| 架构方案 | 主要优势 | 主要不足 |
|------|------|------|
| MVC | 易理解、易上手、适合中小型 Web 项目 | 项目变大后控制流程容易集中，结构扩展性一般 |
| 分层架构 | 职责清晰、便于维护、适合多人协作、与现有仓库结构高度一致 | 如果后续业务非常复杂，可能需要进一步做模块化优化 |
| 微服务架构 | 扩展性强、服务可独立部署、适合大型系统 | 实现和运维复杂度高，对课程项目来说成本过大 |
| 事件驱动架构 | 解耦能力强，适合异步通知和联动扩展 | 调试困难、理解成本高，不适合作为当前主架构 |

### 5.2 与 CampusHub 的适配结论

结合项目目标、仓库现状、团队规模和课程周期，CampusHub 当前阶段最适合采用 **分层架构** 作为主要架构方案。

原因如下：

1. **与当前仓库结构最一致**：后端已有 `controller/service/mapper` 分层骨架，继续沿用成本最低。
2. **适合前后端分工协作**：Vue 前端与 Spring Boot 后端可以围绕明确接口开展并行开发。
3. **适合课程项目节奏**：相比微服务和事件驱动，分层架构更容易在 10 周内稳定落地。
4. **便于后续扩展**：后面新增需求发布、接单、评价、信用等模块时，可以在原有层次上继续扩展。
5. **能保留未来演进空间**：如果后期功能增多，可以在分层架构基础上再逐步引入更强的模块化设计，或在局部场景中引入事件驱动机制。

---

## 6. 本组建议

本组建议在 Phase 2 的最终架构设计中，以 **分层架构** 作为 CampusHub 的主要候选方案结论；同时在候选方案分析中保留 MVC、微服务、事件驱动三类方案作为对照，以体现：

- 我们不仅比较了“能不能做”，也比较了“是否适合当前项目阶段”；
- 我们的选择依据不是方案是否“高级”，而是是否更适合 CampusHub 当前的功能目标、团队协作方式和课程周期。

这样的分析更符合本阶段“架构选择与论证”的任务要求，也更有利于后续 ADR 与架构设计文档的衔接。
