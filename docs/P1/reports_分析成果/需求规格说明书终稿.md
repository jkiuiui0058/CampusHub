## 1. 引言
### 1.1 目的

`[人工补充]`

本文档旨在根据对目标用户的问卷调查和深度访谈结果及ai生成需求说明书补充，整理并定义校园互助服务平台的核心需求。本说明书将作为产品设计、开发和测试的直接依据。

### 1.2 范围

`[AI生成+人工修改]`

本系统为 Web 应用，核心功能包括：  
- 用户注册与身份认证（绑定学号）；  
- 互助需求发布、浏览、搜索与接单；  
- 消息通知与私信沟通；  
- 信用评价与纠纷处理机制；  
- 后台管理（用户审核、内容监管、数据统计）。  

系统不包含支付功能（仅支持用户间自行结算），亦不涉及物流配送调度。

### 1.3 定义

`[AI生成+人工修改]`

#### 术语表  

- 角色类：
  - 用户：指注册并使用本平台的在校学生。
  - 发布者：指在平台上发起互助需求的一方，在具体任务语境下也称需求方。
  - 接单者：指接受并执行互助需求的一方，在具体任务语境下也称接单方。
  - 平台客服：指代表平台处理用户审核、内容监管和争议调解等事务的后台角色。
  - 平台：指本校园互助服务系统本身，注意与平台客服等角色相区分。

- 业务对象类：
  - 互助需求：指用户发布的、可由其他用户承接的服务请求。
  - 需求类型：指互助需求所属的类别，如快递代取、学习辅导、二手交易、组队匹配、失物招领等，不同类型可对应不同的发布入口和处理流程。
  - 用户标签：指用于描述用户特征或身份信息的分类标识，可用于需求匹配与推荐。
  - 信用分：指基于历史完成率、评价结果等信息形成的用户可信度指标。
  - 评价：指任务完成后需求方与接单者双方给出的评分和文字反馈。
  - 凭证：指用于证明任务完成情况的截图、照片或聊天记录等材料。
  - 聊天记录：指用户在平台站内聊天中形成的沟通留痕，可作为纠纷处理依据。
  - 积分：指用户完成任务后获得的平台代币，可用于兑换平台提供的奖励或权益。
  - 虚拟荣誉：指平台提供的非现金、非实物虚拟奖励形式，用于体现用户贡献和成就。


- 业务动作类：
  - 校内认证：指用户通过学校统一身份认证完成注册或登录的过程。
  - 发布需求：指用户按需求类型填写信息并发起互助请求的行为。
  - 内容审核：指平台通过自动或人工方式检查用户发布内容是否存在违法、不良或明显不当信息的过程。
  - 接单：指其他用户接受某项互助需求并进入执行流程的行为。
  - 完成确认：指需求方对接单者提交的任务结果进行确认的行为。
  - 纠纷仲裁：指平台依据聊天记录、凭证等材料对争议任务进行调解、裁决的过程。

- 状态类：
  - 待接单：指需求已发布但尚未被其他用户承接的状态。
  - 进行中：指需求被接单后、尚未完成确认时的任务状态。
  - 已完成：指任务已执行完毕并经需求方确认的状态。
  - 已取消：指需求在执行前或执行中被终止，不再继续处理的状态。

- 规则类：
  - 脱敏显示：指系统在展示学号、手机号等敏感信息时，对完整信息进行部分隐藏或限制公开。
  - 强制站内聊天：指任务相关沟通原则上需在平台内进行，并形成可追溯的记录。
  - 差异化流程：指不同类型和复杂度的需求采用不同的发布、沟通与确认流程。
  - 流量倾斜：指系统根据任务紧急程度、任务地点、用户标签等因素，对部分需求给予更高展示优先级或推荐权重。
  - 多元激励：指平台支持现金、积分、虚拟荣誉等多种回报方式。
  - 小法庭机制：指在客服介入前或调解无效时，由高信用用户依据聊天记录与凭证进行社区投票裁决的纠纷处理机制。
  - 二次确认：指系统在接单等关键操作提交前，要求用户再次确认，以降低误操作风险。
  - CAS：指 Central Authentication Service，即学校统一身份认证系统。

## 2. 总体描述

### 2.1 产品视角

`[AI生成+人工修改]`

本系统是一个面向高校学生的校园互助软件系统，用于承载学生在校园生活中的互助需求。相较于当前主要依赖熟人关系、论坛和群聊完成互助的方式，本系统提供统一的平台载体，用于支持互助信息的组织、展示以及沟通留痕。

本系统与外部环境的主要关系体现在两方面：一是系统运行于校园网络和常见 Web 使用环境中，服务对象为在校学生；二是系统依赖学校统一身份认证机制完成用户身份校验，以建立平台中的基础信任。除身份认证相关外部支持外，系统不承担物流配送调度，也不提供完整的平台内支付闭环。

### 2.2 用户特征

`[AI生成+人工修改]`

本系统的主要用户为在校本科生和研究生，具备基本的 Web 使用能力，能够完成需求发布、信息浏览、接单沟通和结果确认等基本操作。

根据调研结果，目标用户对隐私保护较为敏感，重视身份可信度、历史评价和信用信息，并希望平台在交互上保持轻量、直接和便于快速响应，以适应校园场景中大量即时性、临时性的互助需求。

### 2.3 约束

`[AI生成+人工修改]`

本系统在实现和运行过程中受到以下约束:
- 用户注册与登录需建立在学校统一身份认证机制基础上，以保证平台用户的校内身份真实性；
- 系统需对真实姓名、学号、手机号等敏感信息提供保护，避免在公开页面直接暴露
- 系统以 Web 应用形式提供服务，需要适应校园网络环境和常见浏览器使用场景。

### 2.4 假设

`[AI生成+人工修改]`

本系统的需求建立在以下假设基础上：
- 学校能够持续提供可用的统一身份认证或学号真实性校验支持；
- 校园网络环境能够满足用户日常访问、浏览和沟通的基本需要；
- 目标用户具备基本的线上使用能力，并能够在评价、信用和沟通留痕等机制支持下完成互助协作；
- 当外部认证条件或网络条件发生明显变化时，相关需求内容需要随之调整;
- 项目能通过宣传手段在一定期限内累计到足够数量的用户，使得浏览内容、接单匹配功能能够正常运转

## 3. 功能性需求 (FR)


### 3.1 用户账户与认证

`[AI生成]`

- 编号：FR-001
  - 描述：用户必须通过学校官方统一身份认证（如 CAS）进行注册，以确保用户均为本校学生。
  - 优先级：高
- 编号：FR-002
  - 描述：用户的真实姓名、学号、手机号等敏感信息不得在公开页面展示，仅用于后台认证和紧急联系。
  - 优先级：高

### 3.2 互助需求管理

`[AI生成]`

- 编号：FR-003
  - 描述：平台需预设多种需求类型，如快递代取、学习辅导、二手交易、组队匹配、失物招领等，并为不同类型提供相应的发布入口。
  - 优先级：高
- 编号：FR-004
  - 描述：平台需支持差异化流程设计。简单任务支持快速发布，复杂任务支持更详细的需求描述、多轮沟通和分阶段确认。
  - 优先级：高
- 编号：FR-005
  - 描述：发布者可控制信息公开范围，例如仅显示手机尾号而不公开完整号码，以满足隐私保护需求。
  - 优先级：中

### 3.3 匹配与沟通

`[AI生成+人工修改]`

- 编号：FR-006
  - 描述：系统应根据任务地点、紧急程度和用户标签等信息对需求进行推荐或流量倾斜，减少重要求助信息被淹没的情况。
  - 优先级：中
- 编号：FR-007
  - 描述：所有任务相关沟通应在平台内置聊天工具中进行，并形成全程留痕，作为后续纠纷处理的凭证。
  - 优先级：高

### 3.4 信用与评价体系

`[AI生成]`

- 编号：FR-008
  - 描述：平台需建立基于历史完成率和评价结果的信用分机制，用于展示用户可信度，并为后续权限控制提供依据。
  - 优先级：高
- 编号：FR-009
  - 描述：任务完成后，需求方和接单方均可对对方进行评分和文字评价，形成双向评价机制。
  - 优先级：高

### 3.5 任务完成与纠纷处理

`[AI生成+人工修改]`

- 编号：FR-010
  - 描述：任务完成后应由需求方进行确认。简单任务可通过照片或截图作为凭证，复杂任务可由双方共同确认结果。
  - 优先级：高
- 编号：FR-011
  - 描述：当双方对任务完成状态存在争议时，平台应支持基于聊天记录和任务凭证的纠纷处理，并可引入类似“小法庭”的社区投票仲裁机制作为补充。
  - 优先级：低

### 3.6 激励与收费

`[AI生成]`

- 编号：FR-012
  - 描述：平台应支持现金、积分、虚拟荣誉等多元激励方式，其中积分应具备可兑换实际商品、服务或平台奖励的价值。
  - 优先级：中
- 编号：FR-013
  - 描述：平台可对大额交易或专业服务收取少量手续费，但对小额、日常互助场景应保持免费，以降低使用门槛。
  - 优先级：低

## 4. 非功能性需求 (NFR)

### 4.1 性能

`[AI生成+人工修改]`

- **NFR-001 响应速度**：首页及列表页加载时间应小于 1.5 秒，以保证用户能够快速获取互助信息。
- **NFR-002 并发支持**：系统应支持不少于 5000 名用户并发访问，以满足校园场景下的集中使用需求。
- **NFR-003 高可用**：平台需保证较高的正常运行时间，关键操作在正常使用过程中应具备稳定性和基本保障能力。

### 4.2 安全性

`[AI生成+人工修改]`

- **NFR-004 数据安全**：所有用户敏感数据必须加密存储，并符合国家个人信息保护相关法规要求。
- **NFR-005 内容审核**：平台应建立自动与人工结合的内容审核机制，防止发布违法、不良或明显不当的信息。
- **NFR-006 隐私保护**：学号、手机号等敏感信息在展示时应进行必要脱敏或限制公开范围，避免用户隐私泄露。

### 4.3 可用性

`[AI生成+人工修改]`

- **NFR-007 界面简洁**：系统界面应简洁直观，广告干扰少，核心功能易于访问和理解。
- **NFR-008 关键操作确认**：接单等关键操作应提供必要的二次确认，以降低误操作风险。

### 4.4 可维护性

`[AI生成]`

- **NFR-009 模块化设计**：系统架构应支持未来较为方便地增加新的需求类型或功能模块，以降低后续维护与扩展成本。
