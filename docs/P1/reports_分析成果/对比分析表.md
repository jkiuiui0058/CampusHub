# 对比分析表

| 对比维度                | AI 生成的需求                                                | 人工调研发现的需求                                           | 分析                                                         |
| :---------------------- | :----------------------------------------------------------- | :----------------------------------------------------------- | :----------------------------------------------------------- |
| **AI 正确覆盖的需求**   | - 用户需通过学号认证- 需求分类发布（快递、辅导等）- 建立信用分与评价体系- 内置站内聊天并留痕- 任务完成后由需求方确认- 支持现金、积分等多元激励 | - 强烈要求通过学校统一身份认证（如CAS）- 明确提出需求类型应有差异化流程- 高度重视历史评价、信用分作为信任基础- 认为站内聊天是必须的，且关键约定需留痕- 普遍认为应由需求方确认任务完成- 接受积分激励，但要求其有实际价值 | AI 准确地捕捉到了用户的核心诉求，尤其是在身份认证、信任机制、沟通留痕和激励方式等关键点上，与调研结果高度一致。这表明 AI 在处理通用性较强的校园场景时，能有效模拟出合理的用户需求。 |
| **AI 遗漏的需求**       | - 未强调信息流/推荐算法的重要性- 未提及对“小法庭”式社区仲裁机制的具体构想- 对隐私保护的表述较为笼统 | - 多次提到“帖子被淹没”，希望平台能对紧急/重要需求进行流量倾斜或重新推送- 多位受访者明确提出希望有类似“闲鱼小法庭”的社区投票仲裁机制- 对隐私有非常具体的担忧（如不愿公开宿舍号、手机号），并期望平台提供更细粒度的信息可见性控制（如只显示手机尾号） | AI 的输出偏重于功能模块的完整性，而忽略了用户体验层面的细节。例如，信息过载是现有渠道（如论坛）的主要痛点，但 AI 未将其转化为对平台推荐算法的需求。此外，社区自治的“小法庭”概念是访谈中一个突出的亮点，但 AI 仅将其概括为“客服介入”，丢失了用户期望的社区参与感。 |
| **AI 捏造的需求**       | - 提出“支持 WCAG 2.1 AA 可访问性标准”- 要求“所有 API 调用需 JWT 认证”- 建议“日志保留 ≥ 180 天” | - 调研中完全没有涉及无障碍访问、具体的技术认证方案（JWT）或日志保留时长等技术细节 | 这些是典型的“工程师视角”需求，而非“用户视角”需求。AI 在生成符合 IEEE 830 标准的文档时，倾向于填充一些行业最佳实践或技术规范，但这些内容并未在真实的用户反馈中出现。它们虽然合理，但对于定义产品核心价值而言并非必需，属于过度设计。 |
| **AI 表述有歧义的需求** | - “纠纷仲裁：...可引入类似‘小法庭’的社区投票机制作为辅助。”- “支持上传图片（如二手物品照片）” | - 用户对“小法庭”的理解是：在客服介入前，由高信用度用户基于证据进行投票裁决。- 用户对图片上传有明确场景：简单任务（如代取快递）需要上传签收/现场照片作为完成凭证；复杂任务（如二手交易）才需要上传物品照片。 | AI 将“小法庭”描述为“辅助”手段，弱化了其在用户心中的重要性。同时，AI 将“图片上传”仅与“二手物品”关联，忽略了其在任务完成验证中的普遍性作用。这种表述上的模糊可能导致开发团队误解需求的优先级和应用场景。 |
