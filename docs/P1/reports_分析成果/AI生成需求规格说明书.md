# 校园互助服务平台需求规格说明书  
**（依据 IEEE Std 830-1998 推荐实践）**

---

## 1. 引言

### 1.1 目的  
本文档旨在为“校园互助服务平台”提供完整、清晰、可验证的软件需求规格说明。该平台面向高校大学生，支持用户发布与接单多种类型的互助请求，包括但不限于快递代取、学习辅导、二手交易、组队匹配等。本说明书将作为开发团队、测试人员、项目管理者及最终用户之间达成共识的技术契约，并指导后续系统设计、实现、测试和验收。

### 1.2 文档约定  
- 所有功能性需求以 **FR-XXX** 编号标识；  
- 非功能性需求以 **NFR-XXX** 编号标识；  
- 使用术语表见附录 A；  
- 图表编号采用章节.图序格式（如 3.2.1）。

### 1.3 预期读者与阅读建议  
- **项目经理**：关注整体功能范围与非功能性约束；  
- **开发人员**：重点阅读第3章“具体需求”；  
- **测试工程师**：依据需求条目编写测试用例；  
- **校方/运营方**：关注安全、隐私与合规性要求。

### 1.4 产品范围  
本系统为 Web + 移动端（iOS/Android）混合应用，核心功能包括：  
- 用户注册与身份认证（绑定学号）；  
- 互助需求发布、浏览、搜索与接单；  
- 消息通知与私信沟通；  
- 信用评价与纠纷处理机制；  
- 后台管理（用户审核、内容监管、数据统计）。  

系统不包含支付功能（仅支持线下结算），亦不涉及物流配送调度。

## 2. 总体描述

### 2.1 产品愿景  
构建一个安全、高效、可信的校园内互助生态，降低学生日常事务成本，促进资源共享与社交连接。

### 2.2 用户特征  
- 主要用户：在校本科生、研究生（18–25岁）；  
- 具备基本智能手机操作能力；  
- 对隐私敏感，重视社区信任机制。

### 2.3 运行环境  
- 前端：iOS 13+/Android 10+，主流浏览器（Chrome/Firefox/Safari）；  
- 后端：云服务器（Linux），MySQL/PostgreSQL 数据库；  
- 网络：校园 Wi-Fi / 4G/5G。

### 2.4 设计与实现约束  
- 必须通过学校统一身份认证（如 CAS 或 OAuth2 学号绑定）；  
- 用户数据存储需符合 GDPR 与国内个人信息保护规范；  
- API 接口需支持 RESTful 架构。

### 2.5 假设与依赖  
- 假设校园网络覆盖良好；  
- 依赖学校提供学号真实性验证接口；  
- 假设用户具备基本诚信意识，平台通过评价机制约束不良行为。

## 3. 具体需求

### 3.1 功能性需求

#### 3.1.1 用户账户管理（FR-001）  
- FR-001.1：用户可通过学号+密码或学校统一认证登录；  
- FR-001.2：新用户首次登录需完善真实姓名、学院、手机号（用于紧急联系）；  
- FR-001.3：支持修改个人资料与头像；  
- FR-001.4：支持注销账户（保留历史记录6个月后匿名化）。

#### 3.1.2 互助需求发布（FR-002）  
- FR-002.1：用户可选择需求类型（快递代取、学习辅导、二手交易、组队匹配等）；  
- FR-002.2：每类需求有结构化表单（如快递需填写取件码、地址、时间窗口）；  
- FR-002.3：支持上传图片（如二手物品照片）；  
- FR-002.4：可设置是否公开联系方式或仅通过平台消息沟通。

#### 3.1.3 需求浏览与接单（FR-003）  
- FR-003.1：首页按分类/距离/热度展示需求列表；  
- FR-003.2：支持关键词、标签、时间范围筛选；  
- FR-003.3：点击需求详情页可查看发布者信用分；  
- FR-003.4：接单后双方进入私聊通道，状态变更为“进行中”。

#### 3.1.4 信用与评价系统（FR-004）  
- FR-004.1：每次互助完成后双方可互评（1–5星 + 文字）；  
- FR-004.2：信用分 = 基础分 + 历史评分加权平均；  
- FR-004.3：恶意行为（如爽约、欺诈）可举报，管理员介入冻结账户。

#### 3.1.5 消息与通知（FR-005）  
- FR-005.1：系统推送接单成功、评价提醒等通知；  
- FR-005.2：内置聊天支持文字、图片，消息加密存储7天。

#### 3.1.6 后台管理（FR-006）  
- FR-006.1：管理员可审核高风险需求（如大额二手交易）；  
- FR-006.2：可封禁违规用户并导出操作日志；  
- FR-006.3：提供周报/月报（活跃用户数、需求类型分布等）。

### 3.2 非功能性需求

#### 3.2.1 性能（NFR-001）  
- NFR-001.1：首页加载时间 ≤ 1.5 秒（在校园网环境下）；  
- NFR-001.2：支持并发用户 ≥ 5000 人。

#### 3.2.2 安全性（NFR-002）  
- NFR-002.1：用户密码需加盐哈希存储；  
- NFR-002.2：学号、手机号等敏感信息脱敏显示；  
- NFR-002.3：所有 API 调用需 JWT 认证。

#### 3.2.3 可用性（NFR-003）  
- NFR-003.1：关键操作（如接单）需有二次确认；  
- NFR-003.2：界面符合 WCAG 2.1 AA 可访问性标准。

#### 3.2.4 可维护性（NFR-004）  
- NFR-004.1：模块化设计，便于新增需求类型；  
- NFR-004.2：日志记录关键操作，保留 ≥ 180 天。
