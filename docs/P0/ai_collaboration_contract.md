---

## 第一条：AI 使用原则

- AI 是工具，工程师是责任主体。
- 所有 AI 输出必须经过人工审查后才能合入项目。

---

## 第二条：AI 使用范围

### 允许使用 AI 的场景

- 生成代码片段、单元测试、正则表达式
- 辅助撰写文档（需求文档、设计文档、README 初稿）
- 解释复杂代码逻辑、辅助 Debug
- 生成重复性的配置文件（如 CI 配置、Dockerfile 模板）

### 禁止直接使用 AI 输出（需额外审查）的场景

- 核心业务逻辑算法（必须人工 Review 并理解）
- 数据库 Schema 设计（必须经过架构师评审）
- 订单状态机、信用分计算等关键逻辑

### 完全禁止使用 AI 的场景

- 提交最终的答辩材料或反思日志（必须由团队成员原创撰写）
- 代替团队成员进行 Code Review 的最终决策（AI 建议仅供参考）

---

## 第三条：过程记录要求

- 所有与 AI 的交互 Prompt 必须保留日志（可截图或保存对话链接）
- Git commit message 须标注：
  - `[AI-assisted]`：表示该 commit 包含 AI 生成内容
  - `[Human-written]`：表示该 commit 为人工编写
  - '[Both]': 表示二者皆有参与
- 每阶段结束后，在 `docs/ai_use_reflections/` 目录下填写《AI 协作反思日志》

---

## 第四条：代码合入规则

- AI 生成的代码必须通过至少一名团队成员的 Code Review（通过 GitHub PR 机制）
- 每位成员每个编码阶段至少有 2 个完全手写的核心函数.

---

## 第五条：违约处理

违约记录以阶段为单位统计，每个阶段结束后清零。

| 违规次数 | 处理方式 |
|---------|---------|
| 第 1 次 | 团队内部提醒，违规者需在群内说明原因 |
| 第 2 次 | 负责整理本阶段所有文档 |
| 第 3 次及以上 | 团队重新评估该成员的角色参与度，上报组长或助教协调 |
